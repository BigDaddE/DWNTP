# Network Transport & Encryption Verification

This document records the verification performed to confirm that two DWNTP
hosts — a coordinator (orderer + `peer0` + chaincode + `cli`) and a remote
peer (`peer1`) — communicate over the network using TLS-encrypted channels
rooted in the project's organization-issued certificate authorities, and
that cross-host block replication actually works end-to-end.

## Purpose

The deployment model used in `network/setup_remote_peer.sh` and
`README_REMOTE_NODE.md` runs `peer1` on a physically separate host from
the rest of the network. Every data exchange between the two hosts —
gossip, block delivery, endorsement, chaincode invocation — flows over
the public network boundary between them. This document answers three
questions:

1. Is TLS actually enabled on every endpoint (orderer, peer0, peer1) and
   on every client-initiated connection (from `cli`, from the Rust
   client)?
2. Which certificate authorities anchor that TLS trust, and do they
   correspond to the material generated by `cryptogen` during the
   project's initial setup — rather than, say, a self-signed or
   externally trusted chain?
3. Does cross-host block replication actually occur, producing a
   consistent ledger state across the two hosts?

## Topology Under Test

| Role         | Host               | Containers on that host                                 | Listens on                  |
|--------------|--------------------|---------------------------------------------------------|-----------------------------|
| Coordinator  | `192.168.109.121`  | `orderer.dwntp.com`, `peer0.org1.dwntp.com`, `dwntp-chaincode`, `cli` | `7050`, `7051`, `9999` |
| Remote peer  | `192.168.105.120`  | `peer1.org1.dwntp.com`, `cli`                           | `7061`                      |

Hostname mappings are injected into the remote peer's container at
startup by `start_remote.sh` via `--add-host`, so that TLS hostname
verification succeeds even when the underlying transport is an IP:

```text
orderer.dwntp.com     -> MAIN_PUBLIC_HOST  (192.168.109.121)
peer0.org1.dwntp.com  -> MAIN_PUBLIC_HOST  (192.168.109.121)
dwntp-chaincode       -> MAIN_PUBLIC_HOST  (192.168.109.121)
```

## What We Verified

### 1. TLS is enabled on the remote peer process

Configuration captured from inside the running container:

```bash
docker exec peer1.org1.dwntp.com env | grep CORE_PEER_TLS
```
```text
CORE_PEER_TLS_CERT_FILE=/etc/hyperledger/fabric/tls/server.crt
CORE_PEER_TLS_ENABLED=true
CORE_PEER_TLS_KEY_FILE=/etc/hyperledger/fabric/tls/server.key
CORE_PEER_TLS_ROOTCERT_FILE=/etc/hyperledger/fabric/tls/ca.crt
```

Corresponding startup log lines from `docker logs peer1.org1.dwntp.com`:

```text
INFO [nodeCmd] serve -> Starting peer with TLS enabled
INFO [certmonitor] trackCertExpiration -> The server TLS certificate
     will expire on 2036-04-16 18:05:00 +0000 UTC
INFO [discovery] NewService -> Created with config TLS: true, ...
```

### 2. The TLS server certificate is rooted in the project's real CA

The server certificate that `peer1` presents for inbound TLS connections
was copied out of the running container and inspected:

```bash
docker cp peer1.org1.dwntp.com:/etc/hyperledger/fabric/tls/server.crt .
openssl x509 -in server.crt -noout -subject -issuer -dates -ext subjectAltName
```

```text
subject = CN=peer1.org1.dwntp.com, L=San Francisco, ST=California, C=US
issuer  = CN=tlsca.org1.dwntp.com, O=org1.dwntp.com, L=San Francisco,
          ST=California, C=US
notBefore = Apr 19 18:05:00 2026 GMT
notAfter  = Apr 16 18:05:00 2036 GMT

X509v3 Subject Alternative Name:
  DNS: peer1.org1.dwntp.com, peer1, localhost, peer0.org1.dwntp.com,
       dwntp-chaincode
  IP : 127.0.0.1, 1.2.3.4, 4.3.2.1
```

The issuer is `tlsca.org1.dwntp.com` — the TLS sub-CA generated by
`network/generate.sh`'s `cryptogen` invocation for `org1`. The same file
is mounted into the coordinator's peers, so the two hosts share a common
root of trust anchored in project-local material, not a public CA
bundle.

The remote peer's trusted TLS root (the only CA it will validate
counterparties against):

```bash
docker cp peer1.org1.dwntp.com:/etc/hyperledger/fabric/tls/ca.crt .
openssl x509 -in ca.crt -noout -subject -issuer
```

```text
subject = CN=tlsca.org1.dwntp.com, O=org1.dwntp.com, ...
issuer  = CN=tlsca.org1.dwntp.com, O=org1.dwntp.com, ...
                    ^^ self-signed root, same identity as above
```

The orderer TLS root (shipped into the `cli` bundle for outbound
connections from the remote host to `orderer.dwntp.com:7050`):

```bash
docker exec cli openssl x509 \
  -in /opt/gopath/src/github.com/hyperledger/fabric/peer/crypto/ordererOrganizations/dwntp.com/orderers/orderer.dwntp.com/tls/ca.crt \
  -noout -subject -issuer
```

```text
subject = CN=tlsca.dwntp.com, O=dwntp.com, ...
issuer  = CN=tlsca.dwntp.com, O=dwntp.com, ...
```

Two sibling TLS PKIs — one for `org1`, one for the orderer
organization — as produced by the two-organization `configtx.yaml`
used by the project.

### 3. TLS enforcement is actually active

During an earlier run an external scanner probed `peer1`'s TLS port and
the server rejected the connection because the client did not complete a
proper Fabric/gRPC TLS handshake:

```text
ERRO [core.comm] ServerHandshake -> Server TLS handshake failed in
     92.458µs with error unexpected EOF
     server=PeerServer remoteaddress=192.168.65.1:58453
```

This is direct evidence that `peer1`'s TLS layer is not merely
configured but actively enforced at the socket boundary — a plaintext
gRPC connection to port `7051` would not make it past the handshake.

### 4. Cross-host encrypted block delivery and gossip occurred

The following sequence was captured from `peer1`'s log during an
operational window when the coordinator was online:

```text
INFO [gossip.gossip] New -> Creating gossip service with self membership
     of Endpoint: peer1.org1.dwntp.com:7061, InternalEndpoint:
     peer1.org1.dwntp.com:7051, PKI-ID: cbf6d415e1010dbeeccf676132e0841d...
INFO [gossip.gossip] JoinChan -> Joining gossip network of channel
     dwntpchannel with 1 organizations
INFO [peer.blocksprovider] DeliverBlocks -> Pulling next blocks from
     ordering service channel=dwntpchannel
     orderer-address=orderer.dwntp.com:7050 nextBlock=1
INFO [gossip.privdata] StoreBlock -> Received block [1] from buffer
     channel=dwntpchannel
INFO [gossip.privdata] RetrievePvtdata -> Successfully fetched ... all 1
     eligible collection private write sets for block [1]
     (0 from local cache, 0 from transient store, 1 from other peers)
     channel=dwntpchannel
INFO [gossip.privdata] StoreBlock -> Received block [2] from buffer
INFO [gossip.channel] reportMembershipChanges -> Membership view has
     changed. peers went online: [[peer0.org1.dwntp.com:7051]]
INFO [gossip.privdata] StoreBlock -> Received block [3] from buffer
```

Four facts fall out of these lines:

- `peer1` established a TLS connection to `orderer.dwntp.com:7050` on
  the coordinator host (the hostname-to-IP mapping for that name is the
  coordinator's address, per `start_remote.sh`).
- It then pulled blocks 1, 2, and 3 over that connection.
- It discovered `peer0.org1.dwntp.com:7051` via Fabric's gossip
  membership protocol, which runs over TLS.
- It retrieved private-data collections from `1 from other peers` —
  i.e., it actually opened a second TLS connection (to `peer0`) and
  pulled data across it.

Every one of these transports runs over TLS because `peer1` is
configured with `CORE_PEER_TLS_ENABLED=true` and has no code path for
falling back to plaintext; a plaintext peer or orderer would be rejected
at handshake time as described in §3.

### 5. The ledger is coherent across hosts

A round-trip event was logged from `User2` on the remote host (commit
`19a58b797a5733dcc9e152b985917678d07a4ee43426a15fabfc33b83d60ec21`) and
subsequently queried from the same ledger:

```bash
cargo run --bin dwntp-client -- --user Admin get-all-events
```
```json
[
  {
    "rtu_id": "RTU-A1",
    "event_name": "InternetTest",
    "event_description": "Test over the internet",
    "source_mtu": "VXNlcjFAb3JnMS5kd250cC5jb20=",
    ...
  },
  {
    "rtu_id": "RTU-A1",
    "event_name": "InternetTest",
    "event_description": "Erik - Test over the internet",
    "source_mtu": "VXNlcjJAb3JnMS5kd250cC5jb20=",
    ...
  }
]
```

Base64-decoding the two `source_mtu` values yields
`User1@org1.dwntp.com` and `User2@org1.dwntp.com` — the actual Fabric CID
common names that were used to sign the respective transactions. Since
those transactions were submitted to the orderer over the encrypted
gossip/broadcast path and then delivered to both peers' ledgers, their
presence in a clean post-restart query from the coordinator confirms
end-to-end consistency.

## Reproducible Verification Procedure

The following sequence can be re-run to re-verify the properties above
any time both hosts are online.

### On the remote host

```bash
# A. TLS is enabled, with the expected certificate material
docker exec peer1.org1.dwntp.com env | grep CORE_PEER_TLS
docker cp peer1.org1.dwntp.com:/etc/hyperledger/fabric/tls/server.crt /tmp/p1.crt
docker cp peer1.org1.dwntp.com:/etc/hyperledger/fabric/tls/ca.crt /tmp/p1_ca.crt
openssl x509 -in /tmp/p1.crt   -noout -subject -issuer -dates
openssl x509 -in /tmp/p1_ca.crt -noout -subject -issuer

# B. Outbound TLS handshake against each coordinator endpoint.
# Fabric gRPC endpoints require ALPN h2 and SNI; openssl must be given both.
for PORT in 7050 7051; do
  HOST=$([ $PORT = 7050 ] && echo orderer.dwntp.com || echo peer0.org1.dwntp.com)
  echo "--- $HOST:$PORT ---"
  echo | openssl s_client -connect "$MAIN_PUBLIC_HOST:$PORT" \
                          -servername "$HOST" -alpn h2 2>/dev/null \
    | grep -E '^(Protocol|Cipher|subject|issuer|Verification)'
done

# C. Peer logs show successful cross-host block delivery
docker logs peer1.org1.dwntp.com 2>&1 \
  | grep -E 'TLS enabled|DeliverBlocks|peers went online|Received block|ServerHandshake'

# D. Ledger coherence — an event logged here appears in a Query from elsewhere
cd "<path-to-DWNTP>"
cargo run --bin dwntp-client -- --user User2 log-event \
  --rtu-id "RTU-LINK-TEST" \
  --event-name "EncryptionVerification" \
  --event-desc "cross-host TLS round-trip verification"
cargo run --bin dwntp-client -- --user Admin get-all-events | grep LINK-TEST
```

The expected output for step B is a TLS 1.2 or TLS 1.3 handshake with a
cipher from the recommended profile (e.g.,
`TLS_AES_128_GCM_SHA256`) and a server certificate issued by
`tlsca.org1.dwntp.com` (for the peer) or `tlsca.dwntp.com` (for the
orderer).

### Observing that the wire bytes are encrypted

To remove any doubt that the transport carries ciphertext rather than
plaintext, capture a few packets while generating traffic and observe
that no chaincode argument strings appear in the payload:

```bash
# as root on the remote host, in one shell:
sudo tcpdump -i any -nn -s 0 -w /tmp/fabric.pcap \
  "host $MAIN_PUBLIC_HOST and (port 7050 or port 7051)"

# in another shell, generate a recognizable payload:
cargo run --bin dwntp-client -- --user User2 log-event \
  --rtu-id "RTU-WIRESHARK" \
  --event-name "PlaintextCheck" \
  --event-desc "this-exact-string-should-not-appear-on-the-wire"

# stop tcpdump, then search the capture for the needle:
strings /tmp/fabric.pcap | grep -F "this-exact-string-should-not-appear-on-the-wire" \
  && echo "FAIL: plaintext leak" || echo "PASS: string not found in capture"
```

A properly configured TLS-enforced Fabric network will not contain the
needle string in the pcap; only the TLS record headers, the (plaintext)
SNI hostnames in the ClientHello, and the certificate chain from the
ServerHello will be legible.

## What Is Not Verified Here

- **Mutual TLS (mTLS) between client and peer.** The default Fabric
  configuration uses server-authenticated TLS: the peer's identity is
  verified by the client via the TLS cert chain, but the client is not
  required to present a TLS certificate of its own. Peer/orderer
  authenticate the *caller* by verifying the signed proposal using the
  caller's MSP identity — a separate, independently-verified mechanism
  (see `security_test_forged_identity.md`). Enabling mTLS requires
  setting `CORE_PEER_TLS_CLIENTAUTHREQUIRED=true` and shipping each
  caller a TLS client cert signed by the org TLS CA; this is not
  currently configured for `peer1`.
- **Network privacy of metadata.** TLS encrypts payloads but not
  connection metadata. Which IPs are communicating, on which ports, at
  what times, and at what approximate data rates is observable on the
  path between the hosts.
- **Forward secrecy for already-captured traffic.** Fabric's default
  TLS configuration negotiates ECDHE key exchange, which provides
  forward secrecy for captured traffic, but this document has not
  independently confirmed the negotiated cipher on each connection;
  step B of §Reproducible Verification Procedure does.
- **Denial-of-service surface.** A reachable `7050`/`7051`/`9999` on
  the coordinator and `7061` on the remote is exposed to the internet.
  TLS handshake cost and gRPC-level rate limiting are outside the scope
  of this test.
